# Экзамен: создание CRUD приложения

## Введение

Вам предстоит создать сайт для организации вечеринок (в формате [potluck dinners][]). Пользователи смогут устраивать вечеринки у себя, и присоединяться к другим вечеринкам. Вы будете разрабатывать сайт последовательно, элемент за элементом. Требования к каждому такому элементу описаны в разделе Releases.

---

## Releases

_Пожалуйста, не исполняйте команду `git push` до окончания экзамена_

### Pre-release: Setup

Убедитесь, что в вашем проекте установленные все необходимые вам библиотеки, создайте базу данных.

### Release 0: User Registration

Первое, что нужно сделать - это регистрация пользователей. Не забудьте про шифрование паролей, называйте ваши маршруты по стандартам `REST`, `API`, `CRUD`. [REST API Best Practices
](https://habr.com/post/351890/)

Добавьте ссылку на страницу регистрации в навигационный блок. Реализуйте регистрацию с использованием `JWT-токенов`. `Email` у пользователей должны быть уникальны. Если регистрация не прошла успешно - пользователь должен быть оповещен об этом. Если она прошла успешно - пользователь входит в систему и перенаправляется на домашнюю страницу. Ссылка на страницу регистрации заменяется именем пользователя. Примерно как на Рисунке 1:

### Release 1: Login/Logout

Добавьте `login` и `logout`.
На домашней странице добавьте ссылку на `login` следом за ссылкой на регистрацию. По нажатии на ссылку - пользователя должно перенаправлять на форму авторизации. Пользователи авторизуются при помощи Email и пароля.

Не забудьте обрабатывать ошибки: если у пользователя не получается залогиниться - ему хотелось бы знать причину. При успешном входе - вместо кнопки `login` должна появляться `logout`. При нажатии на `logout` пользователь выходит из системы и оказывается на домашней странице приложения.

### Release 2: Устроить вечеринку!

Добавьте зарегистрированным пользователям возможность устраивать вечеринки. Авторизованные пользователи должны видеть ссылку `устроить вечеринку` на домашней странице. Ссылка должна перенаправлять пользователя на страницу с формой создания вечеринки. На форме пользователь вводит дату, время и место вечеринки.

Когда пользователь нажимает на кнопку отправки формы, то в случае ошибки (валидация?), пользователь должен видеть свою форму, то есть не терять данные в ней, и текст ошибки. Если же все прошло успешно - пользователя перенаправляет на страницу созданной им вечеринки (параметризированная страница), на которой отображаются все подробности вечеринки. Все вечеринки должны быть связанны с пользователями, создавшими их. См. Рисунок 3.

_Примечание:_ Дата и время - не текстовые поля. Посмотрите, какой тип данных лучше подходит для сбора и хранения этих данных.

### Release 3: Просмотр вечеринок.

Добавьте возможность просмотра списка вечеринок. Пусть они отображаются на домашней странице для всех пользователей. Отображать надо только те вечеринки, которые еще не начались (Либо не закончились). Они должны быть отсортированы по времени начала. Самые ближайшие вечеринки (или например те, что идут прямо сейчас) должны быть в самом начале списка. Для каждой вечеринки необходимо отображать название, место, время начала. Имя должно быть ссылкой. По нажатии на нее пользователь переходит на страницу с подробной информацией о соответствующей вечеринке. См. Рисунок 4.

Нажимаем на нее - попадаем на форму редактирования. Форма уже должна содержать текущие данные о вечеринке. Отправка формы должна обновить данные о вечеринке и перенаправить пользователя на страницу вечеринки. В случае ошибки - пользователь остается на форме и видит текст ошибки.

### Release 7: Хочу пойти на эту вечеринку!

Чуть не забыли, зачем затеяли разработку этого приложения. Нам ведь нужно иметь возможность пойти на вечеринку, а не только устраивать их. Пойти могут только авторизованные пользователи.

Отредактируйте страницу вечеринки так, чтобы на ней отображался список всех тех, кто планирует быть на вечеринке. Если пользователь является автором вечеринки - у него не должно быть возможности присоединиться к ней, он итак там будет. Добавьте кнопку с текстом "Хочу пойти!", которая добавит пользователя в список приглашенных на вечеринку. В случае ошибки - пользователь видит сообщение, с указанием, в чем проблема. Если все прошло успешно - пользователь не будет видеть кнопку "Хочу пойти!", вместо этого он будет видеть что-то вроде `${имя пользователя} планирует пойти на эту вечеринку`. См. Рисунок 7.

#### Пример настройки связей и получения данных через алиасы

Для настройки связей между вашими моделями нужно учитывать, что без алиасов нас могут поджидать трудности с получением данных. Без алиасов данные о том, на какие вечеринки пользователь хотел пойти, будут перетирать данные о том, организатором каких вечеринок пользователь является, потому что в обоих случаях ключ будет именован как `Parties`. В таком случае пример связей может быть построен следующим образом:

1. Имеется модель вечеринки, которая может иметь связь с пользователем через атрибут, обозначающий владельца cо специальным алиасом.
2. Каждая вечеринка может иметь много участников; для этого можно задуматься о создании связующей модели c другим специальным алиасом, которая будет хранить информацию о том, кто собирается пойти на эту вечеринку.
3. Модель пользователя тоже должна как-то отличать, автором каких вечеринок он является и на какие вечеринки он хочет пойти.
4. Обратите внимание на то, что алиасы нужно будет использовать и в запросах на получение данных.

Вот пример того, как можно организовать связи и составить запрос на получение пользователей вместе с их вечеринками:

```js
class Party extends Model {
  static associate({ User, PartyUser }) {
    this.belongsTo(User, { foreignKey: 'owner_id', as: 'owner' }); // связь определяющая автора вечеринки
    this.belongsToMany(User, { // связь определяющая приглашенных пользователей на вечеринку
      through: {
        model: PartyUser, // связующая таблица
      },
      as: 'invitedToTheParty', // алиас для приглашенных пользователей на вечеринку
      foreignKey: 'party_id', // ключ для связи
    });
  }
}

class User extends Model {
  static associate({ Party, PartyUser }) {
    this.hasMany(Party, { foreignKey: 'owner_id', as: 'ownedParties' }); // связь определяющая авторство над вечеринками
    this.belongsToMany(Party, { // связь определяющая список вечеринок, на которые собирался пойти пользователь
      through: {
        model: PartyUser,  // связующая таблица
      },
      as: 'partiesAttended', // алиас для списка вечеринок, на которые пользователь собирался пойти
      foreignKey: 'user_id', // ключ для связи
    });
  }
}
```

```js
  class PartyUser extends Model {
    static associate(models) {}
  }
  PartyUser.init(
    {
      user_id: DataTypes.INTEGER,
      party_id: DataTypes.INTEGER,
    },
    {
      sequelize,
      modelName: 'PartyUser',
    }
  );

};
```


```js
const users = await User.findAll({
  include: [
    {
      model: Party,
      as: 'ownedParties', // указываем алиас для получения вечеринок, которыми мы владеем
    },
    {
      model: Party,
      as: 'partiesAttended', // указываем алиас для получения вечеринок, на которые мы ходили или хотим пойти
      through: {
        attributes: ['gift'], // явно указываем получение конкретного атрибута из промежуточной таблицы
        as: 'gift', // указываем алиас для получения данных из промежуточной таблицы, чтобы вывести принесенный с собой подарок
      },
    },
  ],
});
```
