# Exam: Создание учетных записей сотрудников.

### Введение

Тебе предстоит создать приложение по созданию учетных записей сотрудников. Приложение должно позволять создавать, редактировать и удалять учетки, а также просматривать список всех сотрудников добавленных в базу данных.
Добавлением записей занимается администратор ресурса. После того как сотрудник создан, у сотрудника появляется возможность зайти в свой аккаунт и дополнить его информацией.

## Требования к приложению

### Администратор
 - В приложении нет возможности регистрации. Регистрацию сотрудников осуществляет администратор.
 - Учетная запись администратора создается через сиды.
 - Только администратор может создавать, редактировать и удалять учетные записи сотрудников.
 - Только администратор может просматривать список всех сотрудников.
 - При создании учетной записи сотрудника, администратор указывает имя сотрудника, email и пароль, а так же занимаюмую должность.
 - Поле "О себе" заполняется сотрудником после того как он войдет в свой аккаунт.
 - Должность сотрудника выбирается из списка должностей, который создается через сиды.
 - Невозможно добавить должность которой нет в базе данных.

### Сотрудник
- Сотрудник может зайти в свой аккаунт используя email и пароль (который добавил администратор при создании учетной записи).
- После входа в свой аккаунт, сотрудник может дополнить свою учетную запись информацией о себе.
- Сотрудник может видеть и редактировать только свою карточку.

### Release 0: Проектирование базы данных
- В приложении должно быть две модели: `User` и `Position`.
- У модели `User` должны быть следующие поля:
  - `name` - имя сотрудника
  - `email` - email сотрудника
  - `password` - пароль сотрудника
  - `about` - информация о сотруднике
  - `position_id` - идентификатор должности
  - `isAdmin` - булево поле, которое показывает является ли сотрудник администратором.
- У модели `Position` должны быть следующие поля:
  - `name` - название должности

У `User` должны быть следующие валидации:
  - `name` - обязательное поле.
  - `email` - обязательное поле, должно быть уникальным
  - `password` - обязательное поле.
  - `position_id` - обязательное поле.
  - `isAdmin` - обязательное поле.

### Release 3: Создание страницы входа, страницы администратора и страницы сотрудника.
- Создай страницу администратора.
- Создай страницу сотрудника.
- Создай страницу входа в приложение.
- На странице входа должны быть поля для ввода email и пароля.
- После входа в зависимости от того кто вошел в приложение, должна открываться страница администратора или сотрудника.
- Если зашел администратор, то должен происходить редирект на страницу администратора. `/admin`, если сотрудник, то на страницу сотрудника `/user`.
  
### Release 4: Страница администратора
- Открывается по адресу `/admin`.
- Доступна только для администратора.
- Содержит форму для добавления нового сотрудника.
- Содержит список всех сотрудников.
- Содержит кнопку для выхода из аккаунта.
- Содержит кнопку для удаления сотрудника.
- Содержит кнопку для редактирования сотрудника.
- При нажатии на кнопку редактирования, открывается страница с формой редактирования сотрудника `/admin/user/:id`.

### Release 5: Страница сотрудника
- Открывается по адресу `/user`.
- Доступна только для сотрудников.
- Сотрудник может видеть только свою карточку.
- Может дополнить информацию о себе.
  
### Release 6*: Страница сотрудника расширенный функционал
Отобрази на этой странице всех коллег с такой же должностью, как у авторизованного сотрудника. 

### Рефакторинг:
Если основной фукционал реализован, то можно приступать к рефакторингу:
- Страница `/login` должна быть доступна только для неавторизованных пользователей.
- Добавь проверку на api ендпоинты, что бы только пользователь с правами администратора мог создавать, редактировать и удалять сотрудников.
- Если пользователь вводит неправильный email или пароль, то должно выводиться сообщение об ошибке.
