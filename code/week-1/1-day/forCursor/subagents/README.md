# Cursor Subagents - Руководство

Субагенты (Subagents) — это специализированные AI-ассистенты, которым основной Agent может делегировать задачи. Каждый субагент работает в своем context window и специализируется на определенном типе работы.

## 📋 Что такое Subagents?

Субагенты позволяют:
- ✅ **Изоляция контекста** - длинные задачи не засоряют основной conversation
- ✅ **Параллельное выполнение** - запускай несколько субагентов одновременно
- ✅ **Специализация** - кастомные промпты и инструменты для конкретных задач
- ✅ **Переиспользование** - один раз настрой, используй везде

## 🎯 Когда использовать субагенты

| Используй субагенты когда...                          | Используй skills когда...                    |
|------------------------------------------------------|---------------------------------------------|
| Нужна изоляция контекста для долгих задач            | Задача однократная (format, changelog)     |
| Запускаешь несколько задач параллельно               | Нужно быстрое повторяемое действие         |
| Задача требует многошаговой экспертизы               | Задача выполняется в один шаг              |
| Нужна независимая верификация работы                 | Не нужен отдельный context window          |

## 📁 Структура файлов

Субагенты — это **markdown файлы с YAML frontmatter** в папке `.cursor/agents/`:

```
.cursor/
└── agents/
    ├── code-reviewer.md      # Ревьюер кода
    ├── test-generator.md     # Генератор тестов
    ├── doc-writer.md         # Писатель документации
    ├── debugger.md           # Отладчик
    └── verifier.md           # Валидатор работы
```

## 📝 Формат файла

```markdown
---
name: code-reviewer
description: Когда использовать этот субагент
model: fast | inherit | claude-sonnet-4
readonly: true | false
is_background: true | false
---

Промпт для субагента.
Инструкции что и как делать.
```

### Поля конфигурации

| Поле            | Обязательно | Описание                                                   |
|-----------------|-------------|------------------------------------------------------------|
| `name`          | Нет         | Уникальный ID. По умолчанию = имя файла без расширения    |
| `description`   | Нет         | Когда Agent должен использовать этот субагент              |
| `model`         | Нет         | Модель: `fast`, `inherit`, или ID модели. Default: `inherit` |
| `readonly`      | Нет         | Если `true`, субагент работает в read-only режиме          |
| `is_background` | Нет         | Если `true`, субагент работает в фоне                      |

## 🚀 Быстрый старт

### 1. Скопируй примеры в проект

```bash
# Создай папку
mkdir -p .cursor/agents

# Скопируй субагенты
cp forCursor/subagents/*.md <your-project>/.cursor/agents/
```

### 2. Используй субагенты

**Автоматическое делегирование:**
```
Просто попроси выполнить задачу:
"Проверь этот код на безопасность"
→ Agent автоматически делегирует code-reviewer
```

**Явный вызов (slash-синтаксис):**
```
/code-reviewer проверь файл UserService.ts на безопасность
/test-generator создай тесты для calculateTotal
/debugger помоги с ошибкой TypeError
```

**Явный вызов (естественный язык):**
```
Используй debugger субагент чтобы найти причину ошибки
Запусти verifier субагент для проверки что всё работает
```

### 3. Параллельное выполнение

```
Проверь код на безопасность и создай тесты параллельно
→ Agent запустит code-reviewer и test-generator одновременно
```

## 📚 Доступные субагенты

### 1. `code-reviewer.md` - Ревьюер кода

**Когда использовать:**
- Проверка безопасности (SQL injection, XSS, CSRF)
- Анализ производительности (N+1, неоптимальные циклы)
- Проверка читаемости и соответствия best practices

**Вызов:**
```
/code-reviewer проверь src/api/users.ts
```

**Что проверяет:**
- 🔴 Критичные проблемы (безопасность)
- 🟡 Важные проблемы (производительность)
- 🟢 Рекомендации (улучшения)

---

### 2. `test-generator.md` - Генератор тестов

**Когда использовать:**
- Нужны unit тесты для функций
- Нужны integration тесты для API
- Нужно покрытие edge cases

**Вызов:**
```
/test-generator создай тесты для calculateTotal
```

**Что создает:**
- Unit тесты (happy path + edge cases)
- Негативные тесты (error handling)
- Моки для внешних зависимостей
- React Testing Library тесты для компонентов

---

### 3. `doc-writer.md` - Писатель документации

**Когда использовать:**
- Нужны JSDoc/PyDoc комментарии
- Нужен README файл
- Нужна API документация
- Нужны архитектурные описания

**Вызов:**
```
/doc-writer создай API документацию для /api/users endpoint
```

**Что создает:**
- JSDoc комментарии с примерами
- README с установкой и использованием
- API docs с request/response примерами
- Inline комментарии для сложной логики

---

### 4. `debugger.md` - Отладчик

**Когда использовать:**
- Есть ошибка или баг
- Тесты падают
- Неожиданное поведение кода

**Вызов:**
```
/debugger помоги исправить TypeError в UserService
```

**Что делает:**
- Root cause analysis (находит корень проблемы)
- Предлагает минимальный fix
- Добавляет регрессионный тест
- Объясняет профилактику

---

### 5. `verifier.md` - Валидатор (проактивный)

**Когда использовать:**
- После завершения задачи для проверки
- Для независимой верификации работы
- Для проверки что тесты действительно проходят

**Особенность:** `readonly: true` - не может менять код

**Вызов:**
```
/verifier проверь что auth flow действительно работает
```

**Что проверяет:**
- ✅ Код действительно существует
- ✅ Тесты запускаются и проходят
- ✅ Edge cases обработаны
- ✅ End-to-end функциональность работает

## 🎨 Встроенные субагенты

Cursor включает 3 встроенных субагента (не нужно создавать):

| Субагент   | Назначение                          |
|------------|-------------------------------------|
| `explore`  | Поиск и анализ кодбазы              |
| `bash`     | Серии shell команд                  |
| `browser`  | Управление браузером через MCP      |

Agent автоматически использует их когда нужно.

## 💡 Паттерны использования

### Паттерн 1: Verification (Проверка работы)

```
Последовательность:
1. Agent реализует feature
2. Agent делегирует verifier для проверки
3. Verifier проверяет что всё работает
4. Verifier отчитывается о проблемах
5. Agent исправляет проблемы
```

**Польза:** Ловит неполные реализации до production

### Паттерн 2: Orchestrator (Оркестратор)

```
Комплексная задача разбивается на этапы:
1. Planner субагент создает технический план
2. Implementer субагент реализует feature
3. Verifier субагент проверяет реализацию
```

**Польза:** Структурированный подход к сложным задачам

### Паттерн 3: Parallel specialists (Параллельные специалисты)

```
Одновременный запуск:
- code-reviewer проверяет безопасность
- test-generator создает тесты
- doc-writer пишет документацию
```

**Польза:** Экономия времени через параллелизм

## 🔧 Создание кастомного субагента

### Шаг 1: Определи задачу

Спроси себя:
- Это повторяющаяся задача?
- Требует ли она специальной экспертизы?
- Нужна ли изоляция контекста?

### Шаг 2: Создай файл

```bash
# Автоматически через Agent
"Создай субагент security-auditor для проверки auth, payments и sensitive data"

# Или вручную
touch .cursor/agents/security-auditor.md
```

### Шаг 3: Напиши конфигурацию

```markdown
---
name: security-auditor
description: Security specialist. Use proactively when implementing auth, payments, or handling sensitive data.
model: fast
readonly: true
---

You are a security expert.

When invoked:
1. Identify security-sensitive code paths
2. Check for common vulnerabilities
3. Verify secrets are not hardcoded
4. Review input validation

Report by severity: Critical, High, Medium, Low
```

### Шаг 4: Тестируй

```
/security-auditor проверь src/auth/login.ts
```

## 📊 Best Practices

### ✅ Делай

1. **Фокусируй субагенты** - одна четкая ответственность
2. **Пиши хорошие descriptions** - Agent решает делегирование на основе description
3. **Будь конкретным** - избегай общих "helper" агентов
4. **Версионируй** - commit `.cursor/agents/` в git
5. **Используй "use proactively"** в description для автоматического делегирования

### ❌ Не делай

1. **Не создавай десятки субагентов** - начни с 2-3 фокусированных
2. **Не пиши расплывчатые descriptions** - будь специфичным
3. **Не делай длинные промпты** - 2000 слов не делают агента умнее
4. **Не дублируй slash команды** - если задача одношаговая, используй команду


## 📈 Производительность и стоимость

### Trade-offs

| Польза                | Компромисс                           |
|-----------------------|--------------------------------------|
| Изоляция контекста    | Startup overhead на каждый субагент  |
| Параллелизм           | Больше токенов (несколько контекстов)|
| Специализация         | Может быть медленнее для простых задач|

### Когда НЕ использовать

- ❌ Быстрые одношаговые задачи (лучше использовать команды)
- ❌ Когда main agent справится быстрее
- ❌ Когда контекста достаточно в main conversation

## 🔍 Отладка субагентов

### Проверь что субагент загружен

```bash
# Просмотри файлы
ls .cursor/agents/

# Agent видит все .md файлы в этой папке
```

### Проверь description

Description определяет когда Agent делегирует задачу:
- ✅ Хороший: "Use when implementing authentication flows with OAuth providers"
- ❌ Плохой: "Use for general tasks"

### Тестируй явно

```bash
# Вызови явно чтобы проверить работу
/your-subagent выполни простую задачу
```
