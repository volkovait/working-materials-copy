# Пример архитектуры Swarm на LangGraph JS

Этот проект демонстрирует использование официального пакета [`@langchain/langgraph-swarm`](https://reference.langchain.com/javascript/langchain-langgraph-swarm) для построения **децентрализованной** мультиагентной системы, в которой агенты передают управление друг другу напрямую (peer-to-peer), без центрального менеджера.

## Описание примера

Реализована та же утилитарная связка, что и в [`native-supervisor-architecture-example`](../native-supervisor-architecture-example), но с другим принципом маршрутизации:

- **Researcher (Исследователь)** — агент с доступом к чтению файловой системы (`list_directory`, `read_file`). Просматривает директории и читает исходники проекта.
- **Writer (Писатель)** — агент с доступом к записи файлов (`write_file`). Генерирует код или документы и сохраняет их на диск.
- **Handoff-инструменты** — особые инструменты, созданные через `createHandoffTool`. С их помощью каждый агент может передать управление коллеге, когда чувствует, что задача выходит за его специализацию.

В этой архитектуре **нет супервизора**: маршруты выстраивают сами агенты, вызывая handoff-инструменты в нужный момент. LangGraph хранит состояние «последнего активного агента», поэтому при продолжении диалога управление возвращается тому, кто работал последним.

**Требование к агентам:** внутри `createSwarm` вызывается `getHandoffDestinations(agent)`, которому нужен скомпилированный граф LangGraph с методом `getGraph()` и узлом `tools`. Поэтому в примере используются агенты на базе `createReactAgent` из `@langchain/langgraph/prebuilt`. У `createAgent` из пакета `langchain` этого API нет — при запуске возникает ошибка `agent.getGraph is not a function`.

**Пример задачи в коде:** проанализировать `package.json` текущего проекта и сгенерировать на его основе файл отчета `SUMMARY.md`.

## Как запустить

1. Установите зависимости:
   ```bash
   npm install
   ```

2. Убедитесь, что в родительской директории `code/week-4/10-day/node` есть файл `.env` с ключами (если его нет, скопируйте там `.env.example` в `.env`):
   ```bash
   OPENAI_API_KEY=sk-ваш-ключ
   OPENAI_BASE_URL=опциональный_базовый_url
   ```

3. Запустите скрипт:
   ```bash
   npm start
   ```

В консоли будет выведен лог того, как агенты передают управление друг другу. По итогу работы скрипта в папке появится сгенерированный файл `SUMMARY.md`.

## Ключевые элементы кода

### `createHandoffTool`
Создает специальный инструмент, дергая который агент возвращает `Command`, переключающий активного агента в графе. У каждого агента есть свой набор handoff-инструментов — по одному на каждую возможную «цель»:

```javascript
const handoffToWriter = createHandoffTool({
  agentName: "writer",
  description: "Передать управление writer'у, когда вся информация собрана..."
});
```

### `createSwarm`
Собирает агентов в единый `StateGraph`, добавляет общий список сообщений и поле `activeAgent`, а также автоматически связывает узлы по именам handoff-инструментов:

```javascript
const workflow = createSwarm({
  agents: [researcherAgent, writerAgent],
  defaultActiveAgent: "researcher"
});
```

### `MemorySaver` (чекпоинтер)
Обязателен для swarm: он хранит, какой агент был активен последним, и позволяет корректно продолжать многошаговые диалоги. Передается в `workflow.compile({ checkpointer })`.

## Преимущества пакета `@langchain/langgraph-swarm`

- **Минимум шаблонного кода:** не нужно вручную писать узел-маршрутизатор и `addConditionalEdges` — handoff и роутер по активному агенту встроены.
- **Естественная коллаборация:** агенты сами решают, кому передать задачу, на основании своего системного промпта и описаний handoff-инструментов.
- **Память активного агента:** при использовании чекпоинтера система помнит, кто работал последним, и продолжает диалог с ним же.

## Чем Swarm отличается от Supervisor

| Аспект                | Supervisor                                  | Swarm                                          |
| --------------------- | ------------------------------------------- | ---------------------------------------------- |
| Маршрутизация         | Централизованно через супервизора           | Децентрализованно через handoff-инструменты    |
| Кто решает следующего | Отдельный LLM-узел-менеджер                 | Каждый агент сам в нужный момент               |
| Состояние             | История сообщений + следующий узел          | История сообщений + активный агент             |
| Память                | Опционально                                 | Чекпоинтер обязателен для multi-turn           |
| Удобно, когда…        | Нужен строгий контроль и аудит решений      | Агенты работают как равноправная команда       |
